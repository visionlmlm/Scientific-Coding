# 正确工作流示例

## 示例 1：实现一个小功能

用户请求：

```text
给订单页面增加状态筛选。
```

正确的 Agent 行为：

1. 搜索订单页面、已有筛选组件、订单状态类型和 API 查询方式。
2. 确认状态值是否已经在代码或接口 Schema 中定义。
3. 复用已有筛选组件和请求封装。
4. 做小范围修改。
5. 运行相关测试、类型检查或手动验证。
6. 汇报修改文件、验证结果、未验证内容和风险。

推荐回应：

```text
我找到了现有订单筛选组件和订单状态定义，会复用当前筛选组件，并扩展已有查询参数。

完成内容：
- 已给订单页面增加状态筛选。

修改文件：
- ...

验证结果：
- ...

未验证内容：
- ...

潜在风险：
- ...
```

## 示例 2：业务规则不明确

用户请求：

```text
用户不应该看到不可用商品。
```

正确的 Agent 行为：

- 不要自行决定“不可用”是什么意思。
- 先搜索已有商品状态字段和过滤逻辑。
- 询问“不可用”是指无库存、未发布、已禁用、已过期还是区域不可售。

推荐澄清：

```text
我找到了几个可能相关的字段：stock_count、published、disabled_at。
这次需求里的“不可用”具体以哪个字段或规则为准？
```

## 示例 3：重构请求

用户请求：

```text
帮我整理一下支付模块。
```

正确的 Agent 行为：

- 先阅读支付模块。
- 找出具体问题。
- 提出有边界的重构方案。
- 在大范围修改前等待确认。

推荐计划：

```text
我看到三个支付处理器里有重复校验逻辑。
建议范围：
- 只抽取共享校验逻辑；
- 保持对外 API 不变；
- 为抽取后的校验补充聚焦测试。

风险：
- 支付流程敏感，我不会在未确认前改变主流程控制逻辑。
```
