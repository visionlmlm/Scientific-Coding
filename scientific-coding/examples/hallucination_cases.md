# 编造与错误工作流案例

## 错误案例 1：编造 API

错误行为：

```text
Agent 因为觉得名字合理，就调用 client.orders.filterByStatus()。
```

为什么错：

- 该 API 没有从代码、文档、类型或测试中确认。
- 实现可能无法编译，或在运行时失败。

正确做法：

```text
先搜索订单 client 定义、生成的 API 类型和现有订单查询示例，再编写代码。
```

## 错误案例 2：臆想业务规则

错误行为：

```text
Agent 自行决定“非活跃用户”是指 last_login_at 超过 30 天。
```

为什么错：

- 业务阈值必须来自需求、文档、已有代码或用户确认。

正确做法：

```text
请用户确认非活跃规则，或在代码库中找到已有定义后再实现。
```

## 错误案例 3：未经确认的大范围重构

错误行为：

```text
Agent 在修复一个登录按钮小问题时，顺手重写了认证模块。
```

为什么错：

- 修改范围超出用户请求。
- 核心行为未经确认就被改变。
- 风险和验证成本被不必要地放大。

正确做法：

```text
只在最相关文件中修复按钮问题。如果发现重构机会，单独说明，不直接动手。
```

## 错误案例 4：未验证就宣布完成

错误行为：

```text
完成了，代码应该能用。
```

为什么错：

- 没有报告测试、类型检查、构建检查或手动验证。
- Agent 把“可能可用”说成了“已经完成”。

正确做法：

```text
运行可用检查。如果无法运行，说明原因，并列出未验证风险。
```

## 错误案例 5：隐藏不确定性

错误行为：

```text
Agent 默默猜测数据来源，并把猜测当成事实写进实现。
```

为什么错：

- 用户看不到关键未知信息。
- 后续排查会更困难。

正确做法：

```text
明确区分已确认事实、合理推测和待确认问题。
```
