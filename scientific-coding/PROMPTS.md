# 科学 Coding 提示词

## 先理解上下文

```text
请先分析项目结构和相关代码，不要修改任何文件。总结当前实现方式、关键模块、调用链和可复用实现。
```

```text
请先阅读现有代码，不要急着编辑。找出相关文件、已有 API、相似实现和可能的验证命令。
```

## 防止编造接口

```text
请先搜索已有接口定义、类型和调用方式，不要编造 API。只有确认接口真实存在后再实现。
```

```text
不要假设 API、文件、Schema 或业务规则存在。请先在仓库中搜索证据，再开始实现。
```

## 修改前先计划

```text
请先列出修改计划，包括修改文件、修改内容、影响范围、风险和验证方式。等我确认后再改代码。
```

```text
开始编辑前，请给出简洁实现方案：目标文件、实现方式、影响范围、风险和验证方式。
```

## 小范围实现

```text
请按最小范围修改，优先复用已有代码，遵循项目结构和命名规范，不要引入不必要依赖。
```

```text
请用能满足需求的最小改动实现。优先复用已有工具和项目约定，不要扩大修改范围。
```

## 完成后必须验证

```text
完成后请运行相关测试或检查，并说明已验证内容、未验证内容和潜在风险。
```

```text
实现后请运行当前项目中最相关的检查。最后汇报验证结果、未验证部分和剩余风险。
```

## 审查模式

```text
请按科学 Coding 观审查这次修改，重点检查：是否编造接口、是否破坏架构、是否缺少验证、是否遗漏边界和安全风险。
```

```text
请用科学 Coding 观做 Code Review，优先关注编造 API、架构破坏、缺少测试、边界遗漏和安全风险。
```

## 重构保护

```text
如果你认为需要重构，请先说明重构原因、影响范围、风险和验证方案。未经确认不要修改核心逻辑。
```

```text
如果重构是必要的，请先提出方案，不要直接改核心逻辑。方案需要包括原因、范围、风险和验证方式。
```
